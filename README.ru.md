# gemini-claw

[![CI](https://github.com/nicshik/gemini-claw/actions/workflows/preflight.yml/badge.svg)](https://github.com/nicshik/gemini-claw/actions/workflows/preflight.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)

Установка **Antigravity CLI (`agy`)** на сервер OpenClaw и регистрация плагина-пульта
**`antigravity`** для него — под сессией Google **AI Pro**.

🇬🇧 [Read in English](README.md)

> **Кратко** — на хосте OpenClaw, одной строкой от root:
> `curl -fsSL https://raw.githubusercontent.com/nicshik/gemini-claw/main/scripts/bootstrap.sh | sudo bash`
> — скачивает репозиторий и выполняет всю установку (зависимости → preflight →
> install → login → кнопки → рестарт → healthcheck). Единственный ручной шаг —
> OAuth-вход в Google AI Pro. Дальше отправь `/antigravity` в Telegram. Требования
> и пошаговая инструкция ниже.

Google снял отдельный CLI `gemini` для индивидуальных тарифов (он падает с
`IneligibleTierError: UNSUPPORTED_CLIENT → "migrate to Antigravity"`). Преемник —
**Antigravity CLI** (`agy`). Этот репозиторий ставит `agy`, выводит его на `PATH`
гейтвея и устанавливает нативный плагин OpenClaw с пультом `/antigravity` в чате
(Telegram) — без прослоек-переводчиков и без имитации старого CLI `gemini`.

Работает на любом **хосте OpenClaw**, где гейтвей запущен под выделенным
сервисным пользователем. Это связка именно для OpenClaw — плагин ставится через
`openclaw plugins install`, к другим агентским стекам не применим.

## Почему плагин (а не скилл)

Пульт — это **plugin command** OpenClaw, поэтому `/antigravity` обрабатывается *до
LLM-агента*. Навигация по меню, смена модели и статус рисуются кодом за
миллисекунды — модель для отрисовки меню не вызывается, так что пульт мгновенный и
не зависит от медленного или сбоящего бэкенда агента. (Ранняя версия была
агентским *скиллом*: каждое нажатие интерпретировал LLM бота — медленно и с
зависаниями при сбоях модели.) Медленными остаются только реальные вызовы `agy`
(`ask` / `image`) — они медленные потому, что медленный сам `agy`, без надбавки LLM.

Плагин — полноценное переносимое расширение OpenClaw: ставится через
`openclaw plugins install`, живёт в домашней директории сервисного пользователя,
переживает `openclaw update` — и встаёт на любой хост с совместимым OpenClaw без
патчей dist и ручной обвязки.

## Что делает установка

- Ставит `agy` для сервисного пользователя OpenClaw (официальный установщик
  `curl … | bash`).
- Симлинкует `agy` на `PATH` гейтвея (`/usr/local/bin/agy`) — стабильное место,
  переживающее `openclaw update` и смену версии node.
- Ставит хелпер кэша списка моделей (`agy-models`) + ежедневный таймер обновления,
  чтобы меню `/antigravity model` было мгновенным и самообновляющимся.
- Устанавливает плагин **`antigravity`** (`openclaw plugins install`) в
  `~/.openclaw/extensions/`, отдельно от синхронизируемого через git workspace.

## Пульт `/antigravity`

Пользователю ничего настраивать не нужно — всё живёт в Telegram. Отправь
`/antigravity` для пульта, `/antigravity_ask <вопрос>` для вопроса и
`/antigravity_image <описание>` для генерации картинки. Оператор ставит всё один
раз (ниже).

- `/antigravity` — главное меню: кнопки **Модель · Статус · Формат картинок**,
  плюс две команды-действия, напечатанные в теле сообщения моноширинным
  tap-to-copy (`/antigravity_ask` / `/antigravity_image`). Они в тексте, а не на
  кнопках — почему, см. [Модель команд и рендеринга](#модель-команд-и-рендеринга).
- **Модель** → выбор из живого списка моделей (текущая помечена `•`); выбор
  становится `--model` по умолчанию для последующих `ask`/`image`.
- **Статус** → текущая модель по умолчанию и формат картинок.
- **Формат картинок** → постоянный формат (соотношение сторон) по умолчанию для
  `/antigravity_image` (текущий помечен `•`; «авто» сбрасывает). Хранится в том же
  файле состояния, что и выбор модели; переживает рестарты, действует до смены.
- `/antigravity_ask <вопрос>` — разовый промпт, возвращает ответ. Без аргумента
  отвечает моноширинной tap-to-copy подсказкой. Форма-подкоманда
  `/antigravity ask <вопрос>` тоже работает.
- `/antigravity_image <описание>` — генерирует картинку (`generate_image` / Nano
  Banana 2) и присылает её фотографией. Форма `/antigravity image <описание>` тоже
  работает.
  - **Формат (aspect ratio)**: разово — первым словом: `/antigravity_image 16:9
    закат над морем` (принимаются `1:1 2:3 3:2 3:4 4:3 9:16 16:9`), либо
    постоянный дефолт через **Формат картинок** в пульте
    (`/antigravity aspect [значение|auto]`). Плагин не может передать `agy`
    аргументы тула, поэтому формат едет инструкцией в тексте промпта, а
    reasoning-модель заполняет параметр `AspectRatio` тула `generate_image`
    (проверено вживую: `16:9` → 1376×768).
  - **Под каждым фото**: две inline-кнопки. **🔁 Ещё раз** — повторная генерация с
    тем же промптом и форматом, новым сообщением; **✏️ Изменить** — присылает
    полную команду моноширинным tap-to-copy, чтобы поправить и отправить заново.
    Подпись фото тоже содержит tap-to-copy команду (держится ≤1024 символов, чтобы
    кнопки не отклеились от фото). Промпты хранятся по короткому id в
    `~/.gemini/antigravity-image-prompts.json` (callback_data ограничен 64
    байтами; хранятся 100 последних), поэтому кнопки под очень старыми фото могут
    «устареть».
- `/antigravity ping` — быстрая проверка авторизации. `/antigravity reset` —
  сброс модели по умолчанию.

Навигация — edit-in-place: нажатие кнопки переписывает то же сообщение пульта, а
не постит новое (`action.type:"callback"` + `registerInteractiveHandler`,
вызывающий `editMessage`). Текстовые подкоманды остаются как запасной путь.

## Модель команд и рендеринга

Пульт построен вокруг двух неочевидных ограничений Telegram/OpenClaw: две формы
команд (однословная `/antigravity_ask` для автодополнения `/` плюс подкоманды
пульта) и два пути рендеринга с разной силой форматирования (ответ команды умеет
markdown и tap-to-copy, интерактивный edit-in-place — нет). Из этого следует, где
показываются копируемые команды и почему «capture-next-message» не построен.
Подробный разбор с примерами и обходной путь через outbound-адаптер —
[`docs/plugin-internals.md`](docs/plugin-internals.md).

## Требования

Хост должен быть гейтвеем OpenClaw, который ты администрируешь как root:

- **OpenClaw >= 2026.6.11** на гейтвее (SDK плагинов с `registerCommand` /
  `registerInteractiveHandler`). Проверка: `openclaw --version`.
- Гейтвей работает под **выделенным сервисным пользователем** с настоящей домашней
  директорией, под systemd-юнитом. Дефолты скриптов: пользователь `openclaw`, юнит
  `openclaw-gateway`, CLI `/opt/openclaw/bin/openclaw`. На других хостах — через
  переменные окружения (см. *Установка на другой хост*).
- Аккаунт **Google AI Pro** для OAuth agy. Логин **на каждый хост свой** и один раз
  требует браузера (`scripts/login.sh`); токен локален для машины и между
  серверами не копируется.
- **Включённые inline-кнопки** для Telegram-канала — пульт кнопочный, без этого
  кнопки молча не отрисуются:
  ```bash
  openclaw config set channels.telegram.capabilities.inlineButtons dm
  ```
  Установщик этот флаг **не** трогает — это твоя политика канала.
- Инструменты хоста: `bash`, `curl`, `tmux` (только для логина), `systemd`.

## Установка

На целевом сервере, от root — одной строкой:

```bash
curl -fsSL https://raw.githubusercontent.com/nicshik/gemini-claw/main/scripts/bootstrap.sh | sudo bash
```

`bootstrap.sh` скачивает репозиторий (git, или tarball, если git нет) в
`/root/gemini-claw` и запускает `setup.sh`. Флаги setup передаются после `-s --`,
например `… | sudo bash -s -- --yes`. Либо клонируй и запусти сам:

```bash
git clone https://github.com/nicshik/gemini-claw && cd gemini-claw
sudo scripts/setup.sh
```

`setup.sh` оркестрирует весь процесс, идемпотентно и с безопасным перезапуском:
**deps** (доустановка `curl`/`tmux`) → **preflight** (быстрые проверки окружения)
→ **install** (agy + симлинк PATH + плагин) → **login** (OAuth Google AI Pro;
пропускается, если уже авторизован) → **telegram buttons** (включает capability
`inlineButtons`, с бэкапом и `config validate`) → **рестарт гейтвея** →
**healthcheck**. Флаги: `--yes` (не спрашивать перед единственной правкой
конфига), `--skip-login`, `--no-buttons`.

Эквивалент по шагам вручную (то, что выполняет `setup.sh`; удобно для отладки
отдельной стадии):

```bash
sudo scripts/preflight.sh                     # read-only проверки окружения, fail fast
sudo scripts/install.sh                       # agy + симлинк PATH + плагин antigravity + агентские скиллы
sudo scripts/login.sh                         # одноразовый OAuth Google AI Pro (нужен браузер)
openclaw config set channels.telegram.capabilities.inlineButtons dm
sudo RESTART_GATEWAY=1 scripts/install.sh     # рестарт гейтвея для загрузки плагина
sudo scripts/healthcheck.sh                   # проверка всей цепочки
```

Нет исходящего доступа к GitHub с сервера? Скопируй репозиторий:

```bash
scp -r gemini-claw root@SERVER:/root/
ssh root@SERVER 'cd /root/gemini-claw && sudo RESTART_GATEWAY=1 scripts/install.sh'
```

### Установка через ИИ-агента

Если установку ведёт ИИ-агент (Claude Code, Codex, …) с **root-шеллом** на хосте,
дай ему одну команду:

```bash
sudo scripts/setup.sh --yes
```

Она идемпотентна и безопасна для повторного запуска, сама ставит недостающие
пакеты ОС (`curl`, `tmux`) и пропускает уже сделанные шаги. **Единственное, что
агент не может сделать сам, — Google OAuth**: `login.sh` печатает URL, который
*человек* должен открыть в браузере, войти под аккаунтом AI Pro и вставить код
обратно. Агент ведёт всё, кроме этого шага на ~30 секунд — запусти сперва с
`--skip-login`, если логин хочешь сделать отдельно, потом повтори без него.

### Установка на другой хост OpenClaw

Для другого хоста с OpenClaw (к не-OpenClaw агентам плагин не применим) скрипты
параметризованы через переменные окружения — ничего не прибито к конкретному
хосту. Если пользователь/пути гейтвея отличаются от дефолтов — выясни их и передай:

```bash
# сервисный пользователь и openclaw CLI, которые реально использует гейтвей
systemctl show -p User --value openclaw-gateway                               # -> OPENCLAW_USER
command -v openclaw || systemctl show -p ExecStart --value openclaw-gateway   # -> OPENCLAW_BIN
```

Затем тот же процесс с этими значениями (пример — подставь найденное):

```bash
sudo OPENCLAW_USER=openclaw OPENCLAW_BIN=/opt/openclaw/bin/openclaw \
     RESTART_GATEWAY=1 scripts/install.sh
sudo OPENCLAW_USER=openclaw scripts/login.sh                    # OAuth AI Pro на этом хосте (браузер)
sudo OPENCLAW_USER=openclaw OPENCLAW_BIN=/opt/openclaw/bin/openclaw scripts/healthcheck.sh
```

На каждом хосте всё равно нужны: **свежий логин AI Pro** (`login.sh` — OAuth-токен
между хостами не переносится), включённая capability **inlineButtons** для этого
бота и OpenClaw >= 2026.6.11. Там, где раскладка стандартная (пользователь
`openclaw`, CLI `/opt/openclaw/bin/openclaw`), переменные можно не указывать.

### Как проходит логин

`scripts/login.sh` запускает `agy` в tmux-сессии, печатает URL авторизации Google
и ждёт. Открой URL в **локальном** браузере, войди под аккаунтом AI Pro, скопируй
код со страницы и вставь обратно. Скрипт проходит первичный онбординг (тема,
телеметрия, доверие к папке) и проверяет результат. Если
автоматический онбординг не добрался до конца, он печатает команду `tmux attach`
для ручного завершения.

## Обновление живого сервера (git pull, а не scp)

После установки держи хост в синхронизации через **git**, а не повторным
копированием файлов. Неотслеживаемая scp-копия тихо расходится с репозиторием, а
`install.sh` из устаревшей копии охотно переустановит СТАРЫЙ плагин поверх нового
(его самопроверка `cmp` проходит, потому что сравнивает устаревшую копию с тем,
что сама же скопировала). Поэтому держи на хосте настоящий клон:

```bash
# один раз, на сервере (от root):
git clone https://github.com/nicshik/gemini-claw /root/gemini-claw
cd /root/gemini-claw
```

Дальше каждый деплой — это просто:

```bash
cd /root/gemini-claw && git pull
sudo RESTART_GATEWAY=1 scripts/install.sh
sudo scripts/healthcheck.sh
```

Репозиторий публичный, так что обычным `git clone` / `git pull` не нужны ни
креденшалы, ни deploy key.

## Использование `agy` напрямую

Плагин вызывает `agy` как процесс; из шелла им можно пользоваться и напрямую:

- `agy -p "…"` — разовый промпт.
- `agy --model "Gemini 3.1 Pro (High)" -p "…"` — выбор модели (человеческие имена
  из `agy models`; старые id вида `gemini-2.5-pro` молча игнорируются).
- `agy -p "…" --output-format json` — JSON-конверт; ответ в `.response`.
- `cat file | agy -p "Summarize"` — stdin.
- Модели: `Gemini 3.5 Flash (Low|Medium|High)`, `Gemini 3.1 Pro (Low|High)`,
  `Claude Sonnet/Opus 4.6 (Thinking)`, `GPT-OSS 120B (Medium)`.

> `agy` — интерактивный агент: в скриптах всегда закрывай stdin
> (`agy … </dev/null`), иначе он читает остаток stdin как промпт и виснет. Плагин
> и хелпер `agy-models` уже это делают.

## Генерация картинок

В `agy` есть тул `generate_image` (модель **Nano Banana 2**, фиксированная / не
выбирается):

```bash
agy --print-timeout 3m -p 'Use your generate_image tool to create a 1:1 image:
a small matte black cube on a light gray background, no text. Save it as an artifact.'
```

Text-to-image и правка/композиция (до 3 входов); форматы
`1:1 16:9 9:16 4:3 3:4 3:2 2:3`. Оговорки: `--model` не меняет модель картинок;
результат ложится в `~/.gemini/antigravity-cli/brain/<uuid>/…`; генерация
медленнее (поднимай `--print-timeout`) и может быть регионально ограничена в
части eu/us регионов.

**Доставка в Telegram.** OpenClaw отправляет исходящие медиа только из разрешённого
корня (workspace), а не из произвольного локального пути — «сырой» файл из
`~/.gemini/…/brain/…` плагин Telegram отдать не может. После генерации он копирует
свежую картинку из brain в `<workspace>/outputs/antigravity-images/`
(`OPENCLAW_WORKSPACE_DIR` или `~/.openclaw/workspace`) и возвращает как фото *этот*
путь. Без этого отправка молча отклоняется. Если сбоит само копирование — пульт
отвечает понятной ошибкой, а не исключением. Папка сама подчищается до свежайших
`ANTIGRAVITY_IMAGE_KEEP` файлов (по умолчанию 20) при каждой публикации и не
растёт бесконечно.

### Квота картинок (`429 Resource Exhausted`)

Генерация картинок использует **отдельную модель** — Nano Banana 2, id
`gemini-3.1-flash-image` — а Antigravity меряет квоту **по моделям**, поэтому лимит
картинок исчерпывается независимо от текстовых/reasoning-моделей. Можно получить
`429 RESOURCE_EXHAUSTED` на `/antigravity image`, пока у `/antigravity ask` (другая
модель) запаса ещё полно. Плагин это распознаёт и показывает короткое сообщение
(с окном сброса, если `agy` его вернул) вместо сырой ошибки, и никогда не шлёт
устаревшую картинку при сбое.

Оговорка: при 429/503 reasoning-модель иногда «обрабатывает»
ошибку сама — ставит себе таймер повтора, пишет статусный артефакт — и возвращает
**пустой** ответ, так что в stdout `agy` детектировать нечего. Поэтому когда
попытка не дала ни картинки, ни вывода, плагин дополнительно сканирует затронутые
этим запуском транскрипты brain на маркеры `RESOURCE_EXHAUSTED`/`429` и
`MODEL_CAPACITY_EXHAUSTED`/`503` и сразу сообщает настоящую причину, вместо того
чтобы сжечь оставшиеся попытки на «модель не вызвала генератор».

На Google AI Pro квота восстанавливается скользящим окном ~5 часов под недельным
потолком; точный размер корзины картинок и окно сброса Google не публикует, поэтому
цифра «сброс через ~N» приходит живьём с бэкенда. Это **не** фиксированные «100 картинок
в день» и **не** отдельная квота «developer API», отличная от веб-приложения — авторизация
идёт через тот же аккаунт Google AI Pro по OAuth, а у картиночной модели своя корзина
квоты, отдельная от текстовых моделей. Живого
счётчика нет, поэтому любое конкретное число считай непроверенным, если это не окно сброса
из самого `429` от `agy`. Это лимит квоты, а не гео-блок
(региональная блокировка была бы HTTP `403`, а EU/US — поддерживаемые регионы).
Источники: `ai.google.dev/gemini-api/docs/{rate-limits,image-generation}`,
`antigravity.google/docs/plans`, `blog.google` (Nano Banana 2 / лимиты Antigravity).

## Восстановление после `openclaw update`

`agy` и его логин живут в домашней директории сервисного пользователя
(`~/.local/bin/agy`, `~/.gemini/…`); симлинк PATH — в `/usr/local/bin`; плагин — в
`~/.openclaw/extensions/` — ничего из этого обновление OpenClaw не трогает. Если
что-то отвалилось:

```bash
sudo scripts/healthcheck.sh                  # смотри, что сломалось
sudo RESTART_GATEWAY=1 scripts/install.sh    # вернуть симлинк + переустановить плагин + перезагрузить
```

## Файлы

```
plugin/index.js            плагин OpenClaw 'antigravity' (пульт agy)
plugin/openclaw.plugin.json манифест плагина (id, алиасы команд, активация)
                           алиасы: antigravity, antigravity_ask, antigravity_image
plugin/package.json        openclaw.extensions -> ./index.js (без сборки)
bin/agy-models             кэшируемый, самообновляющийся список моделей для пульта

Состояние на сервере (домашняя директория сервисного пользователя, переживает обновления):
~/.gemini/antigravity-skill.json          модель по умолчанию + формат картинок по умолчанию
~/.gemini/antigravity-image-prompts.json  стор промптов для кнопок «Ещё раз»/«Изменить» (100 последних)

scripts/setup.sh           оркестратор одной командой: deps->preflight->install->login->buttons->restart->healthcheck
scripts/preflight.sh       read-only проверки окружения (root, версия openclaw, пользователь, tmux, сеть, диск, кнопки)
scripts/install.sh         идемпотентно: agy + симлинк PATH + установка плагина + агентские скиллы (antigravity_ask, antigravity_image)
scripts/login.sh           одноразовый OAuth Google AI Pro через tmux (сидит конфиг онбординга; key-walk — фолбэк)
scripts/healthcheck.sh     сквозная проверка, ненулевой код выхода при сбое
scripts/uninstall.sh       удалить плагин + симлинк + хелпер (--purge удаляет и agy)
scripts/secret-scan.sh     падает, если какой-то отслеживаемый файл похож на креденшал
docs/plugin-internals.md   заметки для мейнтейнеров: модель команд/рендеринга + внутренности онбординга
docs/onboarding-improvement-plan.md  заметки по дизайну онбординга
.github/workflows/preflight.yml  CI-гейт: синтаксис шелла/плагина, JSON, secret-scan, запрет *bak*
```

## Внутренности онбординга

`login.sh` — единственная капризная часть, потому что у agy **нет подкоманды
`login`/`auth`** — Google OAuth доступен только через интерактивный TUI
(`agy -i`), который `login.sh` держит в tmux, чтобы прочитать auth-URL и вставить
код, который ты вводишь. Устойчивость даёт сидинг конфига онбординга, key-walk как
фолбэк, гейт версии agy и запасной `tmux attach`. Разбор —
[`docs/plugin-internals.md`](docs/plugin-internals.md); дизайн-ретроспектива —
[`docs/onboarding-improvement-plan.md`](docs/onboarding-improvement-plan.md).

## Харденинг

CI (`.github/workflows/preflight.yml`) гейтит каждый пуш: `bash -n` по всем
скриптам и `bin/agy-models`, `node --check plugin/index.js`, валидация JSON
манифестов плагина, запуск `scripts/secret-scan.sh` и запрет отслеживаемых файлов
`*bak*`. Креденшалов в репозитории нет — OAuth-токен AI Pro остаётся в домашней
директории сервисного пользователя на каждом хосте. Тег `v1.1-hardened` — точка
восстановления: если хост потерян, `git checkout v1.1-hardened && sudo
RESTART_GATEWAY=1 scripts/install.sh` пересобирает пульт из заведомо рабочего
состояния (локален для хоста только одноразовый OAuth `scripts/login.sh`).

**Модель доступа.** `/antigravity_ask` (и `ask`/`image` в пульте) передаёт текст
отправителя как есть агенту `agy`, который работает под сервисным пользователем и
имеет собственные инструменты (запись файлов в свой brain/workspace, создание
артефактов). Плагин не добавляет своего контроля доступа сверх авторизации
отправителей в OpenClaw — обработчик кнопок работает по принципу fail-closed, а
команды гейтит сам OpenClaw. Поэтому держите список допущенных собеседников бота
ограниченным (например, `dmPolicy` / pairing): любой, кто может писать боту,
управляет агентом на вашем сервере и тратит вашу квоту AI Pro.

## Безопасность

Кратко: сам вход - это стандартный Google OAuth официальным клиентом `agy`. Код gemini-claw никогда не видит ваш пароль и не трогает долговременный токен. Но безопасность аккаунта сводится к безопасности хоста, и есть реальные оговорки - учитывайте их ниже.

**Кто именно входит.** Через Google логинится только оператор и только один раз на каждый хост (`scripts/login.sh`, браузер нужен один раз). Конечные пользователи Telegram-бота не проходят никакого Google OAuth: их запросы (`/antigravity`, `ask`, `image` и навыки) выполняет уже авторизованный `agy` оператора под сервисным пользователем. Отдельной личности или изоляции у пользователей нет - все допущенные отправители работают под одним аккаунтом AI Pro оператора и тратят одну общую квоту.

**Как устроен вход и чего репозиторий не касается.** Пароль вводится только на странице Google (`accounts.google.com`) - код gemini-claw его не видит. `login.sh` лишь читает URL авторизации, который печатает `agy`, и принимает вставленный вами код авторизации, после чего передаёт его локальному `agy`; обмен кода на токены делает сам `agy`. На запросе "Authorization code" вставляйте именно код авторизации, а не пароль - поле не скрыто и не проверяет, что это код. Долговременный OAuth-токен репозиторий не читает, не копирует и никуда не передаёт - его хранит `agy` в `~/.gemini` сервисного пользователя. В коде плагина и навыков нет ни одного HTTP-клиента и нет скрытой телеметрии: единственный внешний адрес, куда уходит ваш запрос - это Google, через локальный бинарь `agy`; ответы и картинки возвращаются в тот же Telegram-чат отправителя. Телеметрия `agy` по умолчанию выключается сидом в `login.sh` (`enableTelemetry:false`; принудительно - только когда в системе есть python3), и после логина скрипт предупреждает, если `agy` включил её обратно (автоматически не чинит - поправьте вручную). Гейт `scripts/secret-scan.sh` в CI ловит API-ключи, токены GitHub/Telegram и приватные ключи, но это подстраховка, а не полная защита: сам OAuth-токен AI Pro в обычном JSON-виде он не распознаёт, срабатывает он в CI уже после пуша, а не как локальный pre-commit хук, и игнорируемые gitignore или бинарные файлы не проверяет вовсе.

**Реальные риски и что делать.**

- Заведите отдельный Google-аккаунт под это, а не свой основной. Любой, у кого есть root или шелл сервисного пользователя, может прочитать токен из `~/.gemini` и использовать его в пределах выданных прав. Сюда же относится запрос от любого, кому разрешено писать боту: `agy` работает под сервисным пользователем с файловыми инструментами (не только запись, но и чтение) в его домашней директории, поэтому подготовленный запрос от допущенного отправителя может прочитать OAuth-токен под `~/.gemini` и выгрузить его - а не просто потратить квоту. Репозиторий не шифрует токен - защита только на уровне прав файловой системы и хоста. "Machine-local" означает лишь, что репозиторий сам не переносит токен между хостами; это не криптографическая привязка к машине - прочитавший его может использовать токен с любой машины, пока доступ не отозван. При компрометации отзывайте доступ в Google.
- Держите сервер доверенным и односубъектным (single-tenant), под вашим контролем: изоляции между пользователями бота нет, все делят один аккаунт.
- Ограничьте, кто может пользоваться ботом (`dmPolicy` / pairing). Плагин не добавляет своего контроля доступа сверх авторизации отправителей в OpenClaw. Любой допущенный собеседник управляет агентом на вашем сервере, тратит вашу квоту и работает под вашей личностью Google.
- Помните про риск по правилам сервиса. Google различает клиентов по идентичности, и обслуживание многих людей через автоматизацию под одним аккаунтом AI Pro может нарушать условия Google и привести к лимитам или блокировке аккаунта. Набор запрашиваемых OAuth-scope задаёт сам `agy` - оператор его не сужает.
- Прочитайте скрипт установки перед запуском. Установка идёт как root через `curl | sudo bash`: репозиторий тянется с GitHub и выполняется под root, а официальный установщик `agy` тянется с `antigravity.google` и запускается уже под сервисным пользователем, а не под root. Это классический непроверяемый `curl | bash` без сверки контрольной суммы или подписи - доверие держится только на TLS. Просмотрите содержимое скрипта перед запуском.

## Конфигурация (env)

| Переменная | По умолчанию | Смысл |
|---|---|---|
| `OPENCLAW_USER` | `openclaw` | сервисный пользователь гейтвея |
| `OPENCLAW_BIN` | `/opt/openclaw/bin/openclaw` | путь к openclaw CLI |
| `AGY_LINK` | `/usr/local/bin/agy` | куда выведен `agy` на PATH |
| `RESTART_GATEWAY` | `0` | `install.sh`: рестартовать гейтвей для загрузки плагина |

Env рантайма плагина (читается процессом гейтвея, задаётся в окружении гейтвея, а
не `install.sh`): `OPENCLAW_WORKSPACE_DIR` — корень workspace для исходящей
доставки картинок (по умолчанию `~/.openclaw/workspace`);
`ANTIGRAVITY_IMAGE_KEEP` — сколько опубликованных картинок хранить до подчистки
(по умолчанию `20`).

## Примечания

- Креденшалов в репозитории нет; OAuth-токен AI Pro живёт в домашней директории
  сервисного пользователя каждого сервера и никогда не коммитится.
- Требуются: `bash`, `curl`, `tmux` (только логин), CLI `openclaw` и systemd-юнит
  `openclaw-gateway` для перезагрузки плагина.

## Установка через нативный CLI плагинов OpenClaw

Однострочника `openclaw plugins install <этот репозиторий>` нет — и это
сознательно: нативная установка плагина копирует **только** JS плагина. Без
бинарника `agy`, его OAuth-логина Google AI Pro, симлинка `PATH` и хелпера кэша
моделей плагин мёртв — всё это настраивает `scripts/install.sh`. Поэтому
поддерживаемая установка — всегда **клон + `scripts/install.sh`** (bootstrap-скрипт
— норма для плагинов, которые тянут внешний бинарник/сервис). Сам скрипт под
капотом вызывает `openclaw plugins install` для регистрации плагина.
